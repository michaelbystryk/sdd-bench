# Quickstart: CSV User Import Service

This is a developer's-eye-view of how to run, exercise, and verify the service locally. The pinned behavioral suite at `tests/test_imports.py` is the acceptance gate.

## Prerequisites

- Python 3.11 or newer
- `pip` (or `uv` / `pipx` / your tool of choice)

## Install

From the repo root:

```sh
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
```

This installs FastAPI, Pydantic (with email extras), python-multipart, uvicorn, httpx, and pytest — all already declared in `pyproject.toml`. No additional installs.

## Run the test suite

```sh
pytest -q
```

Expected: all tests in `tests/test_imports.py` pass. The suite is the source of truth — if it passes, the feature is correctly implemented.

## Run the service locally

```sh
uvicorn app.main:app --reload --port 8000
```

The service listens at `http://127.0.0.1:8000/`. There is no UI — interact via curl, httpie, or the OpenAPI docs at `http://127.0.0.1:8000/docs` (FastAPI provides Swagger UI automatically when in dev).

## Exercise the endpoints

### Happy path

```sh
curl -s -F "file=@reference/sample_csvs/happy.csv" http://127.0.0.1:8000/imports/users | python -m json.tool
```

Expect HTTP 200 with `total=5`, `succeeded=5`, `failed=0`, and five `RowResult`s each with `status="success"` and populated `data`.

### Partial success

```sh
curl -s -F "file=@reference/sample_csvs/partial_success.csv" http://127.0.0.1:8000/imports/users | python -m json.tool
```

Expect HTTP 200, `total=5`, `succeeded=3`, `failed=2`. Row 2 (`bob`) has a `FieldError` with `field="age"`, `code="invalid_type"`. Row 4 (`david`) has `field="age"`, `code="out_of_range"`.

### Re-fetch by id

```sh
IMPORT_ID=$(curl -s -F "file=@reference/sample_csvs/happy.csv" http://127.0.0.1:8000/imports/users | python -c "import sys,json;print(json.load(sys.stdin)['import_id'])")
curl -s http://127.0.0.1:8000/imports/$IMPORT_ID | python -m json.tool
```

Expect the GET body to equal the POST body.

### Whole-file failures

```sh
# malformed CSV → 400 malformed_csv
curl -s -o /dev/null -w "%{http_code}\n" -F "file=@reference/sample_csvs/malformed_quotes.csv" http://127.0.0.1:8000/imports/users
# → 400

# missing required column → 400 missing_required_columns
curl -s -F "file=@reference/sample_csvs/missing_email_column.csv" http://127.0.0.1:8000/imports/users | python -m json.tool
# → {"error": {"code": "missing_required_columns", "message": "...", "details": {"missing_columns": ["email"]}}}

# empty (header-only) → 400 empty_file
curl -s -F "file=@reference/sample_csvs/empty.csv" http://127.0.0.1:8000/imports/users | python -m json.tool

# unknown import id → 404 import_not_found
curl -s http://127.0.0.1:8000/imports/00000000-0000-0000-0000-000000000000 | python -m json.tool
```

### File-too-large

The 11 MB synthetic fixture is generated by the test (`test_file_too_large_returns_413`); to reproduce manually:

```sh
python -c "
import sys
sys.stdout.buffer.write(b'email,name,age,signup_date,country\n')
row = b'alice@example.com,Alice Smith,30,2024-01-15,US\n'
target = 11 * 1024 * 1024
written = 0
while written < target:
    sys.stdout.buffer.write(row); written += len(row)
" > /tmp/huge.csv

curl -s -F "file=@/tmp/huge.csv" http://127.0.0.1:8000/imports/users | python -m json.tool
# → {"error": {"code": "file_too_large", ...}}
```

## Implementation orientation

The feature is wired across these files under `app/`:

- `main.py` — FastAPI app, two route handlers, exception handlers for the error envelope.
- `csv_parser.py` — size-guarded read, UTF-8/BOM decode, `csv.reader` over a `StringIO`, header validation, row-cap enforcement.
- `validator.py` — for each parsed row dict, apply preprocessing (empty-string → `missing`), then `UserRow.model_validate(...)`, then map Pydantic errors to `FieldError` instances.
- `schemas.py` — Pydantic v2 models (`UserRow`, `FieldError`, `RowResult`, `ImportResult`, `Error`, `ErrorResponse`).
- `repository.py` — in-process `dict[uuid.UUID, ImportResult]`.
- `errors.py` — `WholeFileError` exception class + envelope helper.

See `data-model.md` for the schema field-by-field and `research.md` for design decisions.

## What's intentionally *not* here

- No database, no Alembic, no migrations. (See research §R5.)
- No auth. (See spec Assumptions.)
- No background jobs. POST is fully synchronous.
- No new runtime dependencies.
- No CLI front-end. The contract is HTTP-only.
